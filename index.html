>> Protect your Java applications <<


Why do you absolutely need the pac4j library?

  • Today, in Java applications, handling authentication, authorization and user profile retrieval is very complex: almost every framework and protocol requires a dedicated solution !
  • pac4j, as the Profile and Authentication Client for Java, aims to be a simple and unified security library for all Java frameworks and authentication mechanisms.
One Ring to rule them all...

One client library (pac4j) to rule all authentication mechanisms (OAuth, CAS, OpenID, HTTP, SAML, LDAP, SQL...)
Jérôme Leleu ;-)

What kind of authentication mechanisms is supported by pac4j ?

  • Direct authenticated calls (REST, stateless)
  • Client protocol authentication (stateful, redirection back and forth to an identity provider for login):
    1. from the client application, redirect the user to the "provider" for authentication (mostly HTTP 302)
    2. after a successful authentication, redirect back the user from the "provider" to the client application (HTTP 302) and get the user credentials
    3. with these credentials, get the profile of the authenticated user (direct call from the client application to the "provider").

What authentication mechanisms are actually supported by pac4j ?

  • OAuth (1.0 & 2.0): Facebook, Twitter, Google, Yahoo...
  • CAS: v1.0 / 2.0 / 3.0
  • HTTP: form, basic auth, IP, request parameter
  • OpenID: Google
  • SAML 2.0
  • Google App Engine authentication service
  • OpenID Connect: v1.0
  • LDAP
  • JWT
  • SQL database
  • MongoDB

What projects/environments is pac4j implemented for ?

  • any J2E web application
  • the CAS web SSO project
  • the Play framework 2.x project
  • the Apache Shiro security library
  • the Spring Security library
  • the Ratpack JVM toolkit
  • the Vertx framework
  • the Undertow web server
  • the Sparkjava framework
  • the Jooby micro web framework
