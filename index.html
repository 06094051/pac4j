> pac4j: protect your Java applications <


Why do we absolutely need the pac4j library ?

  • Today, in Java applications, using client side protocols (for authentication and profile retrieval) is very complex: almost every framework and protocol requires a dedicated solution !
  • pac4j, as the Profile and Authentication Client for Java, aims to be a simple and unified library for all Java frameworks and protocols.
One Ring to rule them all...

One client library (pac4j) to rule all protocols (OAuth, CAS, OpenID, HTTP, SAML)...
Jérôme Leleu ;-)

What kind of protocol is supported by pac4j ?

  • It targets all the protocols supporting the following mechanism:
    1. from the client application, redirect the user to the "provider" for authentication (mostly HTTP 302)
    2. after successful authentication, redirect back the user from the "provider" to the client application (HTTP 302) and get the user credentials
    3. with these credentials, get the profile of the authenticated user (direct call from the client application to the "provider").

What protocols are actually supported by pac4j ?

  • OAuth (1.0 & 2.0): DropBox, Facebook, Github, Google, LinkedIn, Twitter, Windows live, WordPress, Yahoo...
  • CAS: protocol versions 1.0 & 2.0 / SAML ticket validation, proxy ticket validation, logout request and CAS proxyfication
  • HTTP: form & basic authentications
  • OpenID: Google
  • SAML 2.0

What projects/environments is pac4j implemented for ?

  • any J2E web application
  • the CAS web SSO project
  • the Apache Shiro security library
  • the Play framework 2.x project
  • the Spring Security library
  • the Ratpack JVM toolkit
